  <name>Task 2: Implement device enumeration + loopback probing</name>
  <files>
src/metamemory/audio/capture/__init__.py
src/metamemory/audio/capture/devices.py
  </files>
  <action>
- Create `src/metamemory/audio/capture/devices.py` with:
  - `list_devices()` returning structured data (dict/list) for debugging.
  - `get_wasapi_hostapi_index()` returning the WASAPI hostapi index when present (by hostapi name contains `WASAPI`).
  - `list_mic_inputs()` that prefers devices on the WASAPI hostapi when available; otherwise all input devices.
  - `list_loopback_outputs()` using a probe strategy designed to avoid false negatives:
    - iterate output devices; for each, attempt to open a short-lived `sounddevice.InputStream` with:
      - `extra_settings=sounddevice.WasapiSettings(loopback=True)`
      - `samplerate=int(device.default_samplerate)` (or a safe fallback like 48000)
      - `channels=min(2, max(1, device.max_output_channels))`
      - `dtype='float32'`
      - a small `blocksize` (e.g., 1024)
    - on success, mark device as `loopback_ok=True`.
    - on failure, capture the exception string as `loopback_error` so the module can print a per-device failure table.
- Provide a module entry point (`python -m metamemory.audio.capture.devices`) that prints a compact summary and exits 0.
- Keep this module side-effect free on import; probing only runs when functions are called.
  </action>
  <verify>python -m metamemory.audio.capture.devices</verify>
  <done>
- Running the module prints at least one device on typical Windows systems.
- Probe failures are handled gracefully (reported, not crash).
- On Windows, WASAPI hostapi is detected and reported when available.
- Loopback probing prints either:
  - at least one `loopback_ok=True` output device, OR
  - a per-device failure table with reasons (so diagnosis is possible).
  </done>
